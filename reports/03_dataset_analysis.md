# 3. Dataset Analysis

## 3.1 GeoLife GPS Dataset

The GeoLife GPS trajectory dataset [Zheng et al., 2009] is a publicly available dataset containing GPS trajectories collected from 182 users over a period of more than five years (April 2007 – August 2012). The full dataset includes 17,621 trajectory files representing over 24 million GPS points with a total recorded distance of approximately 1.2 million kilometres.

### 3.1.1 Dataset Characteristics

| Property | Value |
|---|---|
| Users | 182 individuals |
| Raw trajectory files | 17,621 |
| Total GPS points (raw) | > 24 million |
| Geographic focus | Primarily Beijing, China |
| Sampling rate (typical) | 1–5 seconds, highly irregular |
| Transportation modes | Walking, cycling, bus, car, subway, train, airplane |

### 3.1.2 Data Format

Each trajectory is stored in a `.plt` file:
- Columns: latitude, longitude, 0, altitude, date (YYYY-MM-DD), time (HH:MM:SS)
- First 6 lines contain metadata (skip on load)

## 3.2 Preprocessing

### 3.2.1 Data Cleaning

The following preprocessing steps are applied (implemented in `src/utils/geolife_loader.py` and `src/utils/preprocess_geolife.py`):

1. **Duplicate removal**: Drop rows with identical timestamps within the same trajectory file.
2. **Unrealistic speed filtering**: Remove GPS points implying speeds > 200 m/s (≈ 720 km/h), which indicate sensor errors.
3. **Spatial outlier removal**: Apply Median Absolute Deviation (MAD) with a 5-MAD threshold on step-lengths to remove spatial outliers.
4. **Minimum length filter**: Retain only trajectories with ≥ 100 points after cleaning (configurable; `min_points=100`).
5. **Timestamp sorting**: Sort all points within each file by timestamp.

### 3.2.2 Trajectory Properties Computed

For each cleaned trajectory the following properties are computed:

| Property | Computation |
|---|---|
| Sampling intervals | `Δt_i = timestamp[i] − timestamp[i-1]` in seconds |
| Sampling irregularity | Coefficient of variation: `CV = std(Δt) / mean(Δt)` |
| Instantaneous speed | `v_i = haversine(p_{i-1}, p_i) / Δt_i` (m/s) |
| Segment bearing | Azimuth angle of each segment in degrees [0°, 360°) |
| Direction change | `Δθ_i = min(|θ_i − θ_{i-1}|, 360° − |θ_i − θ_{i-1}|)` |
| Stop regions | Contiguous points with `v_i < 1.0 m/s` for ≥ 30 s |
| Turn events | Points where `Δθ_i ≥ 30°` |
| Total distance | Sum of Haversine step-lengths in km |
| Duration | `timestamp[-1] − timestamp[0]` in minutes |

### 3.2.3 Preprocessing Outputs

Running `python src/utils/preprocess_geolife.py` (default: up to 50 users, `min_points=100`) writes:

| File | Purpose |
|---|---|
| `data/processed/trajectories.pkl` | List of cleaned trajectory `DataFrame`s — **primary input** for `run_experiments.py` and plot scripts |
| `data/processed/trajectory_properties.csv` | One row per trajectory with length, distance, CV, stop/turn counts, etc. |
| `data/processed/trajectories_points.csv` | Long-format GPS table (`trajectory_id`, `user_id`, `file_id`, `point_index`, lat, lon, alt, timestamp) |
| `data/processed/trajectories_index.csv` | Trajectory index (ids and point counts) |

Use `--max-users N`, `--all-users`, or `--export-csv-only` as documented in `reports/11_reproducibility.md`. The notebook `notebooks/01_dataset_analysis.ipynb` can load either the pickle or the CSV exports.

## 3.3 Preprocessed Dataset Statistics

The preprocessed dataset used in all experiments contains **5,716 trajectories** from up to 50 users (subset of the full GeoLife collection, filtered to ≥ 100 points per trajectory). Statistics below are computed on a representative 500-trajectory sample.

### 3.3.1 Trajectory Length Distribution

| Statistic | Value |
|---|---|
| Mean length | 1,437 points |
| Median length | 930 points |
| Std deviation | 1,517 points |
| Minimum | 95 points |
| Maximum | 11,988 points |

The distribution is right-skewed: most trajectories fall in the 200–2,000 point range, but a small number of very long trajectories (> 5,000 points) skew the mean. This wide range requires algorithms to scale efficiently from short to very long trajectories.

![Trajectory Length Distribution](../results/figures/dataset_length_distribution.png)

**Figure 3.1 — Trajectory Length Distribution**

This figure shows three histograms for the 300-trajectory sample: (left) number of GPS points per trajectory, (centre) total distance travelled in km, and (right) trip duration in minutes.

- The length distribution is **right-skewed**: the majority of trajectories contain 200–2,000 points, but a long tail of very long trajectories (>5,000 points) pushes the mean above the median. This skew means any algorithm must handle both very short (under-represented feature-poor) and very long (computation-heavy) trajectories.
- The distance distribution shows most trajectories cover 2–30 km, consistent with urban commutes and day-trips in Beijing.
- The duration histogram confirms that GeoLife trajectories are full daily logs (median ≈ 4 hours), not isolated short trips. This explains why stop-point density is high — users pause frequently throughout the day.

### 3.3.2 Sampling Irregularity

| Statistic | Value |
|---|---|
| Median inter-point interval | 5.0 seconds |
| Mean inter-point interval | 7.3 seconds |
| Std deviation of intervals | 62.4 seconds |
| Mean Coefficient of Variation (CV) | 5.96 |
| Trajectories with CV > 1.0 (high irregularity) | **87.4%** |
| Trajectories with CV < 0.3 (regular sampling) | 6.6% |

A CV of 5.96 (mean) indicates extreme sampling irregularity — far exceeding the CV ≤ 0.3 assumption used by most classical algorithms. The 87.4% of trajectories with CV > 1.0 confirms that **irregular sampling is the norm, not the exception, in real GPS data**. This is caused by GPS signal loss (tunnels, buildings), power-saving modes, and device-specific sampling policies.

**Implication for simplification**: Methods that assume **regular** temporal sampling (or treat all inter-point gaps as equally informative) will systematically mishandle these trajectories.

![Sampling Irregularity](../results/figures/dataset_sampling_irregularity.png)

**Figure 3.2 — Sampling Irregularity**

- **(Left)** Inter-point time interval distribution (clipped at 99th percentile). The sharp spike near 0–5 s reflects the typical GPS recording rate, but the long tail extending to hundreds of seconds shows frequent large gaps caused by GPS signal loss (tunnels, buildings) and power-saving modes.
- **(Right)** Per-trajectory Coefficient of Variation (CV = std/mean of intervals). The red dashed line marks CV = 1.0 — the conventional threshold for "irregular" sampling. **87.4% of trajectories fall to the right of this line**, with a mean CV of 5.96. This is 20× higher than the CV ≤ 0.3 that classical methods implicitly assume.

**Why this matters**: Classical algorithms like Douglas-Peucker treat all inter-point gaps as geometrically equivalent. On these trajectories, a point after a 5-minute gap carries far more unique spatial information than a point in a 1-second burst — but DP has no way to express this distinction. The proposed method's **irregularity score** was designed specifically to address this gap.

### 3.3.3 Distance and Duration Distribution

| Statistic | Distance (km) | Duration (minutes) |
|---|---|---|
| Median | 10.38 | 253 |
| Mean | 32.86 | 346 |

The long median duration (253 minutes ≈ 4.2 hours) reflects that the GeoLife dataset contains full daily GPS logs, not just individual trips. Many trajectories span entire days and include stops, mode changes, and highly variable activity.

### 3.3.4 Speed Distribution

| Statistic | Value |
|---|---|
| Median speed | 1.37 m/s (≈ 4.9 km/h) |
| Mean speed | 3.49 m/s (≈ 12.6 km/h) |
| 99th percentile speed | 31.08 m/s (≈ 112 km/h) |
| Mean fraction of stop points (speed < 1 m/s) | **34.2%** |

More than one-third of all GPS points across trajectories are "stop points" — points where the user is stationary or moving at walking pace or below. This is significantly higher than typical urban vehicle datasets, reflecting the pedestrian/commuter nature of the GeoLife data. Any simplification method that eliminates stop regions for geometric reasons will lose a large fraction of the semantically meaningful data.

![Speed and Stop Statistics](../results/figures/dataset_speed.png)

**Figure 3.3 — Speed Distribution and Stop-Point Density**

- **(Left)** Speed distribution across all GPS point-pairs. The orange vertical line marks the 1 m/s stop threshold. The dominant mass near zero confirms that a very large fraction of all recorded motion is at or below walking/standing speed. The long tail extends to ~30 m/s (fast vehicle/train segments).
- **(Right)** Per-trajectory fraction of stop points (speed < 1 m/s). The mean is 34.2%, meaning that on average **one in three GPS points is a stop point**. No other public GPS dataset we are aware of has this high a stop density, reflecting the mixed-mode, full-day logging nature of GeoLife.

**Implication**: An algorithm that removes geometrically clustered points will systematically delete stop regions (because stationary GPS points are spatially close). The proposed method's **stop score** preserves these regions by scoring them based on duration rather than spatial spread.

### 3.3.5 Turn Distribution

| Statistic | Value |
|---|---|
| Mean fraction of turn points (direction change ≥ 30°) | **32.4%** |
| Median fraction of turn points | 32.1% |

Nearly one-third of all trajectory points represent significant direction changes. In urban environments, these typically correspond to intersections, route decisions, or sudden direction reversals (parking, stairs, etc.). A simplification method that discards these points will produce trajectories that do not represent the actual route taken.

![Turn and Stop Distributions](../results/figures/dataset_turns_stops.png)

**Figure 3.4 — Turn and Stop Point Fraction per Trajectory**

This figure overlays two distributions on the same axis:
- **Purple (turn points)**: fraction of points where direction change ≥ 30°, representing intersections, route decisions, and navigation events.
- **Red (stop points)**: fraction of points where speed < 1 m/s, representing stationary events.

Both distributions peak near 30–35% and are remarkably similar in shape. This means that for a "typical" GeoLife trajectory, roughly **one-third of all points are semantically important** by either the turn or stop criterion — yet neither criterion is captured by geometric distance alone.

The near-equal densities of turns and stops also confirm that neither is a rare edge case that can be ignored; both must be explicitly preserved in any algorithm that claims semantic fidelity. The proposed method handles both via dedicated turn and stop scoring components.

## 3.4 Challenges Identified for Trajectory Simplification

The dataset analysis reveals four concrete challenges that motivate the design of the proposed method:

### 3.4.1 Extreme Sampling Irregularity (CV ≈ 6.0)

With 87.4% of trajectories exhibiting CV > 1.0, interval-based assumptions break down entirely. A point separated from its neighbours by a 10-minute gap holds far more unique spatial information than a point in a dense 1-second sampling burst, yet pure geometric methods treat both equally.

**Proposed response**: The **irregularity score** component explicitly promotes points in sparse regions (large time gaps relative to the local median interval).

### 3.4.2 High Stop Fraction (34.2%)

Over a third of all points are in stop regions. Classical geometric methods (e.g., DP) may aggressively remove stop clusters since the points are geometrically close together, even though these clusters represent significant temporal events (waiting at a bus stop, visiting a location, pausing at a crossing).

**Proposed response**: The **stop score** component explicitly identifies and promotes stop regions based on duration.

### 3.4.3 Dense Turn Events (32.4%)

Nearly a third of all points are at significant direction changes. Since direction changes often occur in small spatial areas (at intersections), geometric distance-based methods may remove them as geometrically redundant, yet they are critical for understanding route decisions.

**Proposed response**: The **turn score** component promotes points with large bearing changes, using smoothing to separate true turns from GPS jitter.

### 3.4.4 GPS Noise in Stop Regions

During stops, GPS coordinates jitter by 5–15 m due to multipath and atmospheric effects. This noise can cause geometric methods to misidentify noisy static points as geometrically important, wasting budget on noise instead of real features.

**Proposed response**: Stop scoring uses duration-based logic (requiring sustained low speed) rather than single-point speed thresholds, making it robust to momentary GPS jitter.

## 3.5 Representative Trajectory Examples

The dataset contains trajectories with the following characteristic patterns relevant to simplification quality:

**Type A — Dense Urban Commute**: Many stops (traffic lights, bus stops), frequent turns at intersections, highly irregular sampling (1–2s in streets, 30s+ in subway tunnels). These trajectories stress stop and turn preservation.

**Type B — Long Walking Session**: Slow speed throughout, many stop events, moderate turn frequency. Irregular sampling due to battery conservation. These trajectories stress irregularity handling.

**Type C — Mixed-Mode Journey**: Starts with walking (slow, high-turn), transitions to high-speed (vehicle), then back to walking. Dramatic speed changes create challenges for uniform-budget allocation.

**Type D — Sparse Long-Distance Trip**: Mostly highway or train travel with long stretches of uniform motion punctuated by rare but important stops and direction changes. These trajectories stress the irregularity score and stop-at-sparse-regions handling.

All four types appear in the preprocessed GeoLife subset, making it an ideal benchmark for evaluating methods across diverse real-world conditions.
